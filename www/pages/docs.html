Documentation

ReMQ is a Redis-backed job queue and scheduler for Deno. Type-safe jobs, full lifecycle (emit, process, retry, DLQ, cron), WebSocket updates, at-least-once delivery, crash recovery, and ~506 jobs/sec full-stack throughput (1.98ms latency).

Changelog — {{REMQ_VERSION}}

At-least-once delivery, crash recovery, RemqManagement API.

Reliability

  • At-least-once delivery — messages stay in PEL until handler completes; ctx.ack() / ctx.nack() are real XACK operations
  • Crash recovery — XCLAIM reclaims messages idle beyond visibilityTimeoutMs
  • visibilityTimeoutMs — configurable (default 30s); set to 3–5× p99 job duration
  • Cron reliability — lock TTL fix, BUSYGROUP handling, correct delay after restart
  • ctx.emitAsync() — awaitable emit; state key and stream entry written in order

RemqManagement

  • RemqAdmin removed — replaced by RemqManagement with management.api.jobs.* and management.api.queues.*
  • Jobs: find(), get(key), delete(key), promote(key), pause(key)
  • Queues: find(), pause(key), resume(key), reset(key), running(key)
  • Events: management.events.job.finished/completed/failed(cb) — all return unsubscribe

New options

  • retryBackoff — 'exponential' | 'fixed' on EmitOptions / HandlerOptions (default fixed)
  • processor.streamPriority — read order across queues (lower = higher priority)

Performance

  • ~506 jobs/sec full lifecycle (~1.98ms avg latency)
  • Trade-off: ACK-after-completion for at-least-once guarantees

Installation

Requires Deno 2.5+ and a running Redis server.

terminal 
deno add jsr:@hushkey/remq

Quick start

main.ts 
import { Redis } from 'npm:ioredis';
import { Remq } from 'jsr:@hushkey/remq';

const opts = { host: '127.0.0.1', port: 6379, db: 1, maxRetriesPerRequest: null };
const db = new Redis(opts);
const streamdb = new Redis({ ...opts, db: opts.db + 1 });

const remq = Remq.create({ db, streamdb, expose: 4000 });
remq.on('my.job', async (ctx) => {
  console.log(ctx.data);
});
remq.start();

const jobId = remq.emit('my.job', { foo: 'bar' });

defineJob

Standalone job files with TApp and TData generics for type-safe context and payload.

jobs/host.sync.ts 
import { defineJob } from 'jsr:@hushkey/remq';

export const hostSync = defineJob<Record<string, unknown>, { hostId: string }>(
  'host.sync',
  async (ctx) => {
    ctx.logger('Syncing ' + ctx.data.hostId);
  }
);
main.ts 
import { Remq } from 'jsr:@hushkey/remq';
import { hostSync } from './jobs/host.sync.ts';
remq.on(hostSync).start();

Remq.create()

Option Type Description
db RedisConnection Redis connection for job storage
streamdb RedisConnection? Optional separate connection for streams
ctx T Context object merged onto handler ctx
concurrency number? Concurrent jobs (default 1)
expose number? WebSocket gateway port
debug boolean? Log worker/job id (default false)
processor.streamMaxLen number? Max stream length per queue (XTRIM)
processor.jobStateTtlSeconds number? TTL for job state keys
processor.maxLogsPerJob number? Max log entries per job
processor.visibilityTimeoutMs number? Visibility timeout for reclaim (default 30000); set to 3–5× p99 job duration
processor.streamPriority Record<string, number>? Read order per queue (lower = higher priority), e.g. { payments: 1, sync: 2 }
processor.pollIntervalMs number? Poll when queue empty (default 3000)
processor.read.count number? XREADGROUP count (default 200)
processor.read.blockMs number? Block time in ms (default 1000)

remq.on()

Accepts (event: string, handler, options?) or a JobDefinition from defineJob(). Returns this for chaining.

Option Type Description
queue string? Queue name (default 'default')
attempts number? Max retry attempts
repeat.pattern string? Cron expression
retryBackoff 'exponential' | 'fixed'? Retry delay strategy (default 'fixed'); exponential = retryDelayMs * 2^retriedAttempts, capped 1h

remq.emit()

Returns jobId: string.

Option Type Description
queue string? Queue name (default 'default')
id string? Custom job id
priority number? Priority
delay Date? Delay until
retryCount number? Max retries
retryDelayMs number? Delay between retries (default 1000)
retryBackoff 'exponential' | 'fixed'? Retry delay strategy (default 'fixed')
repeat.pattern string? Cron expression

remq.start()

Ensures consumer groups, creates the processor, and boots the WebSocket gateway if expose is set.

remq.stop()

Closes the WebSocket server and waits for active jobs to finish.

remq.drain()

Wait for active jobs to finish before shutdown.

Job context

Property / Method Description
id Job id

queue Queue name

retryCount Max retries

RemqManagement setup

management.ts 
import { RemqManagement } from 'jsr:@hushkey/remq';
const management = new RemqManagement({ db, streamdb, remq });

Jobs API

Composite key format: {queue}:{jobId}.

Method Description
management.api.jobs.find() All jobs, deduplicated by jobId (most recent terminal state wins)
management.api.jobs.get(key) Get job by key queue:jobId
management.api.jobs.delete(key) Delete all state keys including all terminal executions
management.api.jobs.promote(key) Fire job immediately, bypass delay
management.api.jobs.pause(key) Set delayUntil to max — job never processes until promoted

Queues API

Method Description
management.api.queues.find() All registered queues including empty streams
management.api.queues.pause(key) Set paused flag; consumer skips the queue
management.api.queues.resume(key) Remove paused flag
management.api.queues.reset(key) Flush all jobs, trim stream, destroy consumer group
management.api.queues.running(key) True if queue is active, false if paused

Events

management.events.job.finished(cb) — completed or failed; .completed(cb) / .failed(cb) for specific status. All return an unsubscribe function.

subscribe.ts 
const unsub = management.events.job.finished((payload) => {
  console.log(payload.jobId, payload.status);
});
// later: unsub();

Retries + DLQ

Configure via Remq.create({ processor: { retry: {...}, dlq: {...} } }) and/or remq.on(..., { attempts }).

Level Option Description
processor.retry maxRetries Max retries (default 0)
processor.retry retryDelayMs Delay between retries (default 1000)
EmitOptions / HandlerOptions retryBackoff 'exponential' | 'fixed' (default 'fixed'); exponential = retryDelayMs × 2^attempts, capped 1h
processor.retry shouldRetry (error, attempt) => boolean
processor.dlq streamKey DLQ stream key
processor.dlq shouldSendToDLQ (message, error, attempts) => boolean

Both attempts > 0 (on handler/emit) and processor.retry.maxRetries > 0 are required for retries.

example 
const remq = Remq.create({
  db, streamdb,
  processor: {
    retry: { maxRetries: 3, retryDelayMs: 2000 },
    dlq: { streamKey: 'dlq' },
  },
});
remq.on('job', handler, { attempts: 3 });

Cron jobs

Use repeat.pattern (cron expression) in remq.on(event, handler, { repeat: { pattern: '0 * * * *' } }). On restart, cron is deduplicated (single fire per pattern). 

remq.on('hourly.cleanup', handler, {
  repeat: { pattern: '0 * * * *'  },
});

WebSocket

Server: set expose: port in Remq.create. In handlers use ctx.socket.update(data, progress) to push updates.

Client: connect to ws://host:port, send emit to enqueue, receive job_update / job_finished. 

// Server
ctx.socket.update({ step: 2, progress: 50 });

// Client: connect, then send { type: 'emit', event: 'my.job', data: {} }
// Receive { type: 'job_update', jobId, data } and { type: 'job_finished', jobId, status }

App context

Type your app context and pass it to Remq.create<AppCtx>({ ctx: { db, mailer } }). Handlers receive ctx.db, ctx.mailer, etc. 

type AppCtx = { db: Redis; mailer: Mailer };
const remq = Remq.create<AppCtx>({ db, streamdb, ctx: { db, mailer } });
remq.on('send.email', async (ctx) => {
  await ctx.mailer.send(ctx.data);
});
