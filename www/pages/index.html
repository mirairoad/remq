import { Remq } from 'jsr:@hushkey/remq';

Benchmarks

Full lifecycle (emit → process → state → retry/DLQ). Pick a scenario:

  • Single
  • Async

Single instance

Hardware: MacBook Pro M1 Pro (10-core, 16GB) · Redis: OSS 7 (Docker) · Concurrency: 10 workers

Throughput: 506 jobs/sec
Latency: 1.98ms avg
Round trips: 2 Redis ops/job
Full lifecycle: emit → queue → process → ack → trim. At-least-once delivery.

Async (emitAsync)

Hardware: MacBook Pro M1 Pro (10-core, 16GB) · Redis: OSS 7 (Docker) · Concurrency: 10 workers

Throughput: ~450 jobs/sec
Latency: ~2.2ms avg
Guarantee: State key and stream entry written in order before return
Use for: payments, invoices, critical job chains
Trade-off: sequential writes vs fire-and-forget emit()

Cluster (3× Mac Mini M4)

Hardware: 3× Mac Mini M4 (10-core, 16GB) — projected · Redis: OSS 7 + Replica (dedicated node) · Concurrency: 10 workers per instance (30 total)

Throughput: ~1,500 jobs/sec (projected, linear scaling from 506/sec single)
Latency: ~1.98ms avg

Features

defineJob

Type-safe standalone job files, TData inference.

Retry + DLQ

Per-job retry, shouldRetry filter, auto DLQ routing.

Cron scheduler

repeat.pattern, dedup on restart.

WebSocket gateway

ctx.socket.update(), per-job or broadcast.

RemqManagement API

Jobs & queues API, job events (finished/completed/failed).

Queue stats

Per-status counts, dashboard-ready.

App context

TApp generic, services/db/mailer typed on ctx.

Tokio runtime

True parallel I/O, not single event loop.

Self-cleaning

TTL, MAXLEN, maxLogsPerJob, no manual cleanup.
